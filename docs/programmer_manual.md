# Руководство программиста CRM-системы «Тихий час»

Настоящее руководство разработано в соответствии с требованиями ГОСТ 19.504-79 и предназначено для программистов, системных администраторов и специалистов по сопровождению программного обеспечения. Документ содержит сведения, необходимые для понимания архитектуры, настройки, развертывания и доработки программных модулей CRM-системы сети барбершопов «Тихий час».

## 1 Назначение и условия применения программы

### 1.1 Назначение программы
Программный комплекс предназначен для автоматизации бизнес-процессов сети барбершопов. Система обеспечивает:
- ведение базы данных клиентов, мастеров, услуг и филиалов;
- онлайн-запись клиентов и управление расписанием мастеров;
- сбор аналитических данных по выручке и загрузке филиалов;
- управление ролевым доступом (Клиент, Мастер, Администратор, Владелец).

### 1.2 Условия применения
Для корректного функционирования серверной части программного комплекса необходима следующая программно-аппаратная среда:
- **Операционная система:** Windows, Linux или macOS (рекомендуется Ubuntu 22.04 LTS или Windows Server 2022).
- **Платформа выполнения:** .NET 8.0 Runtime (или SDK для разработки/сборки).
- **СУБД:** SQLite (встроенный провайдер, внешняя СУБД не требуется).
- **Аппаратные требования (минимальные):** 1 ядро CPU, 1 ГБ ОЗУ, 5 ГБ свободного дискового пространства.

## 2 Характеристики программы

### 2.1 Архитектура и структура
Программный комплекс реализован как монолитное веб-приложение с использованием архитектурного паттерна Clean Architecture (упрощенный вариант). Решение (`DiplomSpa.sln`) разделено на три основных проекта:

1. `BarbershopCrm.Domain` — содержит бизнес-сущности (POCO-классы) и перечисления. Не имеет внешних зависимостей.
2. `BarbershopCrm.Infrastructure` — отвечает за доступ к данным (Entity Framework Core), миграции базы данных, механизмы аутентификации (собственная реализация PBKDF2-HMAC-SHA256 без Identity) и загрузку изображений.
3. `BarbershopCrm.Web` — точка входа в приложение. Содержит слой представления на базе ASP.NET Core Razor Pages, статические файлы (CSS, скрипты, изображения), конфигурацию DI-контейнера и `Program.cs`.

### 2.2 Используемые технологии и библиотеки
- Фреймворк: ASP.NET Core 8.0 (Razor Pages).
- ORM: Entity Framework Core 8.0 + Microsoft.Data.Sqlite.
- Логирование: Serilog (с записью в консоль и файлы с ротацией).
- Тестирование: xUnit, FluentAssertions.

## 3 Обращение к программе

### 3.1 Запуск и сборка проекта
Для сборки проекта и подготовки артефактов для развертывания используется интерфейс командной строки .NET CLI.
Сборка и восстановление зависимостей осуществляются командой:
```bash
dotnet restore
dotnet build --configuration Release
```
Запуск проекта в режиме разработки:
```bash
dotnet run --project src/BarbershopCrm.Web
```
В среде разработки (`Development`) приложение автоматически применяет миграции и создает файл базы данных `barbershop.db` в каталоге запуска.

### 3.2 Настройка и конфигурация
Основные настройки программы хранятся в файле `appsettings.json`.
Для изменения параметров приложения необходимо отредактировать данный файл:
- `ConnectionStrings:DefaultConnection` — путь к файлу БД SQLite.
- `Serilog` — параметры логирования (уровни логирования, пути к файлам логов).
- `Auth` — настройки сессий и куки-файлов.

### 3.3 Работа с базой данных
Управление схемой базы данных осуществляется через Entity Framework Core Tools.
Для создания новой миграции после изменения доменных моделей:
```bash
dotnet ef migrations add <ИмяМиграции> --project src/BarbershopCrm.Infrastructure --startup-project src/BarbershopCrm.Web
```
Для обновления базы данных (в Production-среде применяется вручную):
```bash
dotnet ef database update --project src/BarbershopCrm.Infrastructure --startup-project src/BarbershopCrm.Web
```

## 4 Входные и выходные данные

### 4.1 Организация базы данных
Данные хранятся в реляционной СУБД SQLite. Основные сущности:
- `Users`, `Roles`, `UserSessions` — подсистема аутентификации и сессий.
- `Branches`, `Services`, `Masters` — справочники (каталог).
- `WorkSchedules` — график работы мастеров.
- `Bookings`, `Visits`, `Leads` — записи, визиты и заявки (основной процесс).
Целостность данных обеспечивается внешними ключами и ограничениями базы данных (например, уникальный частичный индекс для защиты от двойного бронирования одного и того же слота).

### 4.2 Обработка изображений
Система принимает на вход файлы изображений (аватары мастеров, фото филиалов). Обработка загрузки (`ImageUploadService`) осуществляет:
- проверку расширения файла;
- генерацию уникального имени;
- сохранение в директорию `wwwroot/images/uploads`.
На выход система отдает относительный URL-адрес сохраненного изображения для отображения в браузере.

## 5 Сообщения

Программа формирует различные типы сообщений для информирования пользователей и системных администраторов.

### 5.1 Пользовательские сообщения
Сообщения для пользователя (валидация форм, успешное сохранение, ошибки бизнес-логики, например, «Слот уже занят») выводятся в интерфейсе веб-страницы через механизм TempData (всплывающие alert-блоки) и `ModelState` (сообщения об ошибках под полями ввода).

### 5.2 Системные сообщения и логирование
Системные ошибки, предупреждения и информационные сообщения (аудит авторизации, исключения БД) обрабатываются библиотекой Serilog.
Правила маршрутизации логов:
- Информационные сообщения (создание записей, логины) выводятся в консоль сервера.
- Предупреждения и ошибки (Exceptions) записываются в файлы журнала, расположенные в папке `logs/` (формат `app-YYYYMMDD.log`).
Ротация файлов настроена на ежедневное создание нового файла с ограничением хранения за последние 14 дней. Анализ файлов журналов является основным методом диагностики при возникновении сбоев в работе приложения.
